Проверка отключения CSM

CSM (Compatibility Support Module) — это режим совместимости, который позволяет использовать старый BIOS-режим загрузки вместе с UEFI.

Проверка в BIOS/UEFI:

  • 1 Найдите параметр CSM/Legacy Mode/Compatibility Support Module
  • 2 Убедитесь, что он отключен (Disabled)

Возможные проблемы при включенном CSM:

  • Некорректная установка операционной системы
  • Проблемы с созданием EFI раздела
  • Возможны сбои в работе загрузчика
Проверка включения UEFI режима

UEFI — это современный стандарт загрузки, который заменяет старый BIOS. Для работы с UEFI-загрузчиками (например, GRUB) этот режим должен быть включен.

Проверка режима загрузки:

$ [ -d /sys/firmware/efi ] && echo "UEFI" || echo "BIOS"

UEFI

Ожидаемый результат: UEFI

Последствия неправильной настройки UEFI:

  • Возможны ошибки загрузки операционной системы
  • Не будет работать безопасная загрузка (Secure Boot)
  • Система может загружаться в Legacy режиме вместо UEFI
Проверка настройки TPM

TPM (Trusted Platform Module) — это криптопроцессор, который обеспечивает дополнительный уровень безопасности системы.

Проверка загрузки модуля TPM:

$ lsmod | grep tpm

tpm_tis                 16384  0
tpm                    49152  1 tpm_tis
tpm_tis_core          32768  1 tpm_tis

Возможные проблемы при отсутствии TPM:

  • Невозможность использования полного шифрования диска
  • Снижение общего уровня безопасности системы
Проверка отключения VT-d

VT-d (Virtualization Technology for Directed I/O) — это технология виртуализации прямого ввода-вывода. Иногда она может вызывать конфликты с определенным программным обеспечением или драйверами.

Проверка в BIOS/UEFI:

  • 1 В настройках BIOS найдите раздел Advanced или CPU Configuration
  • 2 Найдите параметр Intel VT-d или AMD IOMMU
  • 3 Убедитесь, что он отключен (Disabled)

Важность отключения VT-d:

  • Помогает избежать конфликтов с драйверами ядра Linux
  • Улучшает стабильность работы RAID массивов
  • Предотвращает проблемы с шифрованием

├─sda2        8:2    0    50G  0 part /

  • Файловые системы должны соответствовать ожидаемым типам
  • Проверьте наличие достаточного свободного места
Проверка RAID1 массива для /boot/efi

RAID1 для /boot/efi обеспечивает отказоустойчивость загрузочного раздела системы.

Проверка состояния RAID массивов:

$ cat /proc/mdstat

Personalities : [raid1]
md4 : active raid1 sdb4[0] sdc4[1]
      975360 blocks super 1.2 [2/2] [UU]

Проверка файловой системы:

$ lsblk -f

NAME    FSTYPE   LABEL UUID
md4     vfat           UUID-HERE    /boot/efi

Проверка монтирования:

$ mount | grep efi

/dev/md4 on /boot/efi type vfat (rw,relatime)

На что обратить внимание:

  • Статус [UU] означает, что оба диска в массиве активны
  • Файловая система должна быть vfat для совместимости с UEFI
Проверка LUKS шифрования

LUKS (Linux Unified Key Setup) обеспечивает шифрование разделов. Важно убедиться, что все разделы правильно зашифрованы и функционируют.

Проверка статуса шифрования:

$ sudo cryptsetup status sda2_crypt

/dev/mapper/sda2_crypt is active and is in use.
  type:    LUKS1
  cipher:  aes-xts-plain64
  keysize: 512 bits
  device:  /dev/sda2

Проверка всех шифрованных устройств:

$ sudo dmsetup ls --tree

sda2_crypt (253:0)
├─vg-root (253:1)
│ └─-
└─vg-swap_1 (253:2)
  └─-
sda4_crypt (253:3)
└─home (253:4)
  └─-

Важные моменты проверки:

  • Все критичные разделы должны быть зашифрованы
  • Проверьте правильность типа шифрования и размера ключа
  • Убедитесь, что все устройства активны и используются

Инструкции по настройке

Изменение символической ссылки в Linux

Цель

Изменить существующую символьную ссылку /usr/local/httpd/htdocs для указания на директорию /usr/share/aster вместо текущей /usr/share/nginx/html/htdocs

Требования

  • Доступ root или права sudo
  • Существующая символьная ссылка /usr/local/httpd/htdocs
  • Наличие директории /usr/share/aster

Шаг 1: Проверка существования целевой директории

ls -ld /usr/share/aster
drwxr-xr-x 2 root root 4096 Oct 10 12:34 /usr/share/aster

Шаг 2: Изменение символической ссылки

sudo ln -snf /usr/share/aster /usr/local/httpd/htdocs

Шаг 3: Проверка параметров команды

-s: Создать символическую ссылку
-n: Не следовать существующей ссылке при её удалении
-f: Принудительно удалить уже существующую ссылку

Шаг 4: Проверка результата

ls -l /usr/local/httpd/htdocs
lrwxrwxrwx 1 root root 16 Oct 10 12:35 /usr/local/httpd/htdocs -> /usr/share/aster

Особые случаи

Если директория /usr/share/aster отсутствует

Перед созданием ссылки убедитесь, что целевая директория существует. Если нет, создайте её с помощью команды:

sudo mkdir -p /usr/share/aster

Важные замечания

  • Убедитесь, что у вас есть необходимые права доступа для изменения ссылки
  • При использовании параметра -f будьте внимательны, так как это приведёт к безвозвратному удалению существующей ссылки
  • После изменения ссылки проверьте работоспособность вашего сервера
  • Рекомендуется сделать резервную копию текущей конфигурации перед внесением изменений

Монтирование разделов

Цель

Правильное монтирование разделов с учетом файловой системы, прав доступа и параметров монтирования

Требования

  • Права суперпользователя (root)
  • Знание типа файловой системы раздела
  • UUID или метка раздела (опционально)

Шаг 1: Подготовка точки монтирования

mkdir -p /mount/point
chmod 755 /mount/point

Шаг 2: Определение UUID раздела

blkid
lsblk -f

Шаг 3: Временное монтирование

mount /dev/sdX1 /mount/point
mount UUID="your-uuid-here" /mount/point
mount -t ext4 -o defaults,noatime /dev/sdX1 /mount/point

Шаг 4: Постоянное монтирование (fstab)

cp /etc/fstab /etc/fstab.backup
nano /etc/fstab
# Add to /etc/fstab: UUID=your-uuid-here /mount/point ext4 defaults,noatime 0 2

Шаг 5: Проверка и применение

mount -a
df -h

Специальные случаи

Монтирование RAID массива
mount /dev/md0 /mount/point
Монтирование зашифрованного раздела
cryptsetup luksOpen /dev/sdX1 cryptname
mount /dev/mapper/cryptname /mount/point

Важные замечания

  • Всегда создавайте резервную копию fstab перед внесением изменений
  • Проверяйте права доступа к точке монтирования
  • Используйте UUID вместо имен устройств для надежности
  • После изменения fstab проверяйте корректность с помощью mount -a

Настройка RAID1 для EFI раздела

Цель

Настройка отказоустойчивого EFI раздела с использованием RAID1 для повышения надежности загрузки системы

Требования

  • Установленная система Debian
  • Два или более диска с разметкой GPT
  • Поддержка UEFI в BIOS
  • Права суперпользователя (root)

Шаг 1: Подготовка системы

apt update
apt install mdadm efibootmgr
Reading package lists... Done Building dependency tree... Done Reading state information... Done
lsblk
fdisk -l

Шаг 2: Создание RAID1 массива

mdadm --create --verbose /dev/md4 --level=1 --raid-devices=2 --metadata=1.0 /dev/sdb4 /dev/sdc4

Шаг 3: Форматирование и монтирование RAID массива

mkfs.vfat /dev/md4
mkdir -p /boot/tmp
cp -a /boot/efi/* /boot/tmp/
umount /boot/efi
mount /dev/md4 /boot/efi
cp -a /boot/tmp/* /boot/efi/

Шаг 4: Обновление конфигурации системы

lsblk -o +uuid
mcedit /etc/fstab
UUID=your-uuid-here /boot/efi vfat umask=0077 0 1

Шаг 5: Настройка RAID конфигурации

mdadm --examine --scan | grep metadata=1.0 >> /etc/mdadm/mdadm.conf
update-initramfs -u

Шаг 6: Настройка загрузчика

efibootmgr -v
efibootmgr -b XXXX -B
grub-install --target=x86_64-efi --efi-directory=/boot/efi --bootloader-id=debian --recheck /dev/sdb
grub-install --target=x86_64-efi --efi-directory=/boot/efi --bootloader-id=debian --recheck /dev/sdc
update-grub

Проверка работоспособности

1. Проверка статуса RAID
cat /proc/mdstat
Personalities : [raid1] md4 : active raid1 sdb4[0] sdc4[1] 976768 blocks super 1.0 [2/2] [UU]
2. Проверка монтирования
mount | grep efi
3. Проверка загрузочных записей
efibootmgr -v

Важные замечания

  • Всегда делайте резервную копию данных перед работой с разделами
  • Проверяйте UUID разделов перед добавлением в fstab
  • Убедитесь, что оба диска поддерживают GPT и UEFI

Настройка KMS (Kernel Mode Setting) для Intel

Цель

Настройка Kernel Mode Setting для корректной работы видеовывода Intel в Linux

Требования

  • Видеокарта Intel
  • Права root
  • Текстовый редактор (nano/vim)

Шаг 1: Редактирование конфигурации GRUB

sudo nano /etc/default/grub
# Найдите строку:

# Измените её на:
GRUB_CMDLINE_LINUX_DEFAULT="quiet splash i915.modeset=1"

Шаг 2: Обновление конфигурации GRUB

sudo update-grub

Как работает KMS vs VBE

VBE (Video BIOS Extensions)
  • BIOS/UEFI использует VBE для базовой настройки экрана
  • Устаревшая технология
  • Может "забыть" о видеовыводе при отсутствии монитора
  • Редко поддерживается современными видеокартами
KMS (Kernel Mode Setting)
  • Ядро Linux напрямую управляет видеокартой
  • Сохраняет конфигурацию даже при отключении монитора
  • Автоматически активирует видеовывод при подключении
  • Работает со всеми современными видеокартами

Совместимость

  • Intel: Параметр i915.modeset=1 работает со всеми современными картами
  • NVIDIA: Использует драйвер nouveau, параметр не применяется
  • AMD: Использует драйвер amdgpu, параметр не применяется
  • Гибридная графика: Параметр работает только для Intel-части

Почему KMS лучше?

  • Работает со всеми современными видеокартами
  • Не зависит от BIOS/VBE
  • Более стабильное и надёжное управление графикой
  • Поддерживает горячее подключение мониторов
